### [HIGH] [security] Unbounded rooms map and connection count — memory-exhaustion DoS
- **Where:** `backend/internal/relay/relay.go:192-214, 122-146, 69-76`
- **Issue:** join() creates a *room for any distinct room id on demand (rooms[roomID] = &room{}) with no global cap on the number of rooms, and serveConn accepts every upgrade with no per-IP or global connection limit. A room is only deleted when it becomes empty (leave, line 235). An attacker can open one connection each to a huge number of distinct random room ids (the id is attacker-chosen — any non-empty string is accepted, line 102-106), each leaving a half-occupied room and a live goroutine + peer struct resident in RAM.
- **Impact:** Unbounded RAM and goroutine growth on a RAM-only service with no DB to offload to — the relay can be OOM-killed / pod-evicted (kind/GKE), taking down pairing for all users. No authentication gates room creation.
- **Fix:** Add a global room/connection cap under r.mu and reject when exceeded. Minimal: in join(), before rm = &room{} for a new id, check len(r.rooms) against a configurable maxRooms and return a "full"/over-capacity signal so serveConn closes the connection with a status code (e.g. a new const or reuse StatusRoomFull) instead of allocating. Optionally add a per-IP counter map keyed by clientIP(req) decremented in leave(). Also set p.conn.SetReadLimit on each connection to bound per-frame memory. Pair with an ingress/LB connection limit as defense-in-depth, but the in-process cap is the load-bearing fix since the gce L7 ingress provides none by default.
- **Verifier evidence:** Attack path: attacker (any network client reaching the public relay) opens N WebSocket connections to /ws?room=<unique-random-id>, one distinct id per connection, and holds them open. Each connection: (1) inserts a new *room+*peer into rooms (relay.go:195-197, 203), never deleted because the room never goes empty while the conn is held (leave deletes only at len==0, 235-237); (2) leaves a live read loop (pump, 145) + keepalive goroutine (143), each goroutine stack + coder/websocket read/write buffers + kernel TCP buffers = several KB resident. With the hard 128Mi limit (infra/base/relay-deployment.yaml:33) on a single replica (replicas: 1, line 9) and no DB to offload, on the order of tens of thousands of idle conns exhausts RAM and the pod is OOM-killed / evicted, dropping all live pairings; restart loses all room state (RAM-only). Grep confirming no mitigation exists: `rg -n -i "limit|rate|maxconn|connlimit|SetReadLimit" backend/ infra/` returns only resource.limits in the deployment yaml and unrelated doc/comment hits — no connection or room cap, no per-IP map, no read limit. No authentication gates room creation (Handler only checks roomID != "", relay.go:102-106).

### [HIGH] [correctness] ExpiresInS never enforced by the agent — no hard timeout, Ask blocks forever
- **Where:** `backend/internal/agent/agent.go:165-213`
- **Issue:** req.ExpiresInS is marshalled into the sealed request and sent to the phone (mcp.go:185, ask.go:68) but the Ask re-announce loop's only termination conditions are ctx.Err() (agent.go:181-183, 199-201, 208-210) and a non-resend error. No context.WithTimeout/WithDeadline is ever derived from ExpiresInS anywhere in the codebase (verified by grep: only the phone UI counts it down in frontend). In `serve` and `ask` the ctx is signal.NotifyContext(Background) (main.go:28) — it never fires absent SIGINT. So when the phone is offline/never answers, Ask retries with capped backoff forever and the MCP tool call (and the one-shot `ask`) hangs indefinitely.
- **Impact:** The documented invariant 'hard timeout => error never approved' is not met. A pending request with expires_in_s=300 that the human ignores blocks the agent/LLM tool call permanently rather than returning ErrTimeout after the expiry. ErrTimeout (agent.go:41) is effectively unreachable in the deployed binaries except via Ctrl-C. The phone's countdown reaching 0 and the agent's view diverge: the phone shows 'expired' while the agent still waits.
- **Fix:** In Ask, before the loop, derive a deadline from req.ExpiresInS so ctx cancellation produces the already-wired ErrTimeout: `if req.ExpiresInS > 0 { var cancel context.CancelFunc; ctx, cancel = context.WithTimeout(ctx, time.Duration(req.ExpiresInS)*time.Second); defer cancel() }`. The existing ctx.Err()->ErrTimeout branches (agent.go:181-183, 199-201, 208-210, 238-239) then turn the expiry into ErrTimeout automatically, keeping the 'never approved on failure' invariant intact. Optionally add a table test: paired agent, persistent SignalUndeliverable, req.ExpiresInS small -> require.ErrorIs(err, ErrTimeout) and nil Approved (mirroring agent_test.go:220-233 but driving timeout via ExpiresInS instead of the test ctx).
- **Verifier evidence:** grep (rg -n 'WithTimeout|WithDeadline|ExpiresInS' backend --type go): ExpiresInS appears only at wire.go:177-178 (def), wire_test.go:72, mcp.go:25/185, ask.go:68 — zero WithTimeout/WithDeadline references it. Repro: call request_approval with expires_in_s=300, then never answer on the phone. relay returns peer-absent -> askOnce returns errResend (agent.go:244) -> Ask hits agent.go:206 (best-effort Notify) then sleep(ctx, backoff) (agent.go:208) and loops; with ctx = signal.NotifyContext(Background, Interrupt) (main.go:28) the only exit is SIGINT. After 300s the phone shows 'expired' but Ask is still looping; ErrTimeout (agent.go:41) never returned. agent_test.go:228 shows the existing timeout test supplies the deadline externally, not via ExpiresInS — no test covers the ExpiresInS path.

### [HIGH] [correctness] Concurrent request_approval calls race on the shared Session connection
- **Where:** `backend/internal/agent/agent.go:165-168, 226-266, 306-314`
- **Issue:** Ask reads a.sess under a.mu once (agent.go:166-168) then operates on the returned *Session with no lock. askOnce calls writeEnvelope/readEnvelope directly on sess.conn (agent.go:231,236) and reconnect mutates sess.conn = conn (agent.go:312) without holding a.mu. The MCP go-sdk dispatches tool calls concurrently (no per-server serialization in requestApproval, mcp.go:169). Two concurrent request_approval calls share the same *Session/sess.conn: their writes interleave on one WebSocket, both goroutines read from the same frameConn, and reconnect in one goroutine swaps sess.conn out from under the other — a data race on the sess.conn field plus frame cross-delivery.
- **Impact:** Data race (detectable under -race) on sess.conn. Worse for correctness: goroutine A can consume goroutine B's decision frame; decodeDecision filters by id (agent.go:296) so a mismatched frame is dropped (continue), but that means B's decision is silently lost and B re-announces / waits forever, while interleaved writes can corrupt the relay's room-of-two framing. The relay being a dumb forwarder, two concurrent senders in the same room break the request/response correlation entirely.
- **Fix:** Serialize Ask with a single-flight guard matching the one-human/one-card model. Add to Agent: `asking atomic.Bool` (or a dedicated mutex). At the top of Ask (agent.go:165), after fetching sess: `if !a.asking.CompareAndSwap(false, true) { return wire.Decision{}, errors.New("agent: another request is in flight") }; defer a.asking.Store(false)`. This makes one outstanding request at a time, eliminates the sess.conn data race and frame cross-delivery, and surfaces a clear error to the MCP client instead of silently losing a decision. Add a unit test that launches two concurrent Ask calls and asserts exactly one proceeds (run with -race). If concurrent requests must be supported later, the upgrade path is per-request rooms/connections rather than a shared sess.conn.
- **Verifier evidence:** SDK proof of concurrency: $GOMODCACHE/github.com/modelcontextprotocol/go-sdk@v1.6.0/mcp/server.go:1444 `if req.IsCall() && req.Method != methodInitialize { jsonrpc2.Async(ctx) }`; dispatch loop /Users/.../go-sdk@v1.6.0/internal/jsonrpc2/conn.go:640-645 (go func runs handler; <-releaser.ch released early by Async at conn.go:336). Repro/attack path: an MCP client (Cursor/Claude) issues two request_approval tool calls without awaiting the first (clients may parallelize tool calls). Both handlers reach h.ag.Ask concurrently -> both read the same a.sess -> askOnce A and askOnce B both writeEnvelope on sess.conn (agent.go:231) and readEnvelope on sess.conn (agent.go:236). Frames interleave on the single WebSocket to the dumb relay. The phone replies to A; the relay forwards it; whichever goroutine's readEnvelope wins gets the frame. If B's read consumes A's decision, decodeDecision(plain, B.id) returns ok=false (agent.go:296 dec.ID != id) -> continue -> A's decision is dropped, A waits/re-announces. Meanwhile reconnect (triggered by a transport error in either goroutine) executes sess.conn = conn (agent.go:312) while the other goroutine is mid writeEnvelope/readEnvelope on sess.conn -> data race on the sess.conn field (would fire go test -race). frameConn (wsConn, conn.go:28-48) has no internal mutex, so concurrent Write calls are also unsafe per coder/websocket (one writer at a time).

### [HIGH] [correctness] Transient reconnect silently drops an open request card; user can never answer, agent waits forever
- **Where:** `frontend/src/lib/session.ts:218-238, 258-281`
- **Issue:** onBox() adds every shown request id to seenIDs (line 273). onConnState(closed) forces screen to 'offline' but leaves state.request intact (line 233). onConnState(open) then sets screen to 'listening' (line 224: offline -> listening), while state.request is still the unanswered request. renderScreen only shows the card when state.screen is yesno/choice/text, so the pending card is now hidden behind the listening screen. The agent (per the resilience design) re-announces the same request id, but onBox de-dupes it (line 272 seenIDs.has) and never reopens the card. Net: a brief relay/network blip while a card is open permanently hides that request from the user.
- **Impact:** A single transient WebSocket reconnect during an open yes/no/choice/text card makes the request unanswerable on the phone for its entire lifetime. The agent keeps waiting (correctly never auto-approves), but the human can never act, so the request effectively dead-locks until it expires. Matches the threat model's 'reconnect & resume' and 'duplicate/dedupe' failure class.
- **Fix:** On reconnect, restore the open card instead of landing on listening. In onConnState('open') (session.ts:220-229), when paired and a request is still pending, route to its card screen:

```ts
if (this.state.paired) {
  let screen = this.state.screen;
  if (screen === 'offline') {
    screen = this.state.request
      ? (this.state.request.response.kind === 'choice' ? 'choice'
        : this.state.request.response.kind === 'text' ? 'text' : 'yesno')
      : 'listening';
  }
  this.set({ conn, attempt, screen });
} else {
  this.set({ conn, attempt });
}
```

This reuses the same kind->screen mapping already in onBox (session.ts:274). Add a regression test mirroring session.test.ts:202-212 but opening a yesno card before ws.close(), asserting screen returns to 'yesno' (not 'listening') after reconnect. The alternative (defer seenIDs.add from onBox to sendDecision) also works but weakens de-dupe for genuinely re-shown-but-still-open cards, so the screen-restore fix is preferred and minimal.
- **Verifier evidence:** session.ts:233 `const screen = this.state.paired && this.state.screen !== 'pair' ? 'offline' : this.state.screen;` converts the open 'yesno'/'choice'/'text' screen to 'offline' on a drop while keeping state.request. session.ts:224 `const screen = this.state.screen === 'offline' ? 'listening' : this.state.screen;` then maps 'offline' -> 'listening' on reconnect. session.ts:272-273 de-dupes the agent's re-announce by req.id. App.tsx:242-243 renders ListeningScreen for 'listening' (ignores state.request). agent.go:174+185 reuse the same req.ID across re-announces (errReconnect at agent.go:232/241 triggers reconnect+resend). No test exercises drop-during-open-card, so it is uncovered.

### [HIGH] [privacy] Yes/No decision leaks via ciphertext length (no padding)
- **Where:** `frontend/src/lib/wire.ts:166-179`
- **Issue:** A yesno Decision is serialized as JSON {"kind":"decision","id":"...","result":{"approved":true|false}} (encodeDecision, newYesNoDecision) and sealed with secretbox, which adds no padding (crypto.ts seal/sealWithNonce -> nacl.secretbox; Go side identical). The wire frame is base64(nonce(24)||plaintext+16). The literal `true` (4 bytes) vs `false` (5 bytes) makes the approve frame exactly one plaintext byte shorter than the decline frame, everything else (id length, keys, nonce, tag) constant for a given request. The relay forwards each frame verbatim (relay.go:165 writeRaw) and trivially sees its byte length.
- **Impact:** The relay is the untrusted network attacker in this threat model. By measuring the size of the single phone->agent decision frame it can distinguish APPROVE from DECLINE for every yesno request without breaking any crypto. This defeats the content-blind confidentiality claim for the most security-relevant bit in the system. Choice/text answers similarly leak coarse length; yesno is a clean 1-bit oracle.
- **Fix:** Pad the sealed app plaintext to a fixed length before seal() on BOTH sides so approve/decline (and ideally all decisions of a kind) are byte-identical. Minimal: in encodeDecision (wire.ts:167) and the Go EncodeDecision path, after JSON-encoding, right-pad with a whitespace/filler run to a fixed size (e.g. next multiple of 256), or add a "_pad" filler string field sized to a constant target. Whitespace padding survives JSON.parse on both sides with no schema change; a _pad field is cleaner but must be ignored by decoders. Apply identically to request frames to also kill choice/text coarse-length leakage. Add one assertion/test that seal(approve-decision).length === seal(decline-decision).length for the same id.
- **Verifier evidence:** Attack path: malicious/compromised relay in a room of two. (1) Phone sends one decision frame agent-bound after a yesno request. (2) Relay (relay.go:152 p.conn.Read) measures len(data) of that frame. (3) For a fixed request id, the only variable in the JSON plaintext is the literal true(4)/false(5); secretbox/base64 are length-transparent, so the approve frame is reproducibly shorter than the decline frame by one plaintext byte. (4) Relay reads the human's approve/decline decision without breaking secretbox or SPAKE2. Grep proof of no mitigation: rg -n "pad|filler|fixed.?len|block" over backend+frontend returns only payload.ts base64url padding, theme.ts pad2, push.ts web-push padding, and CSS padding — none touch the sealed decision plaintext.

### [HIGH] [privacy] QR pairing payload (?p=) carries the SPAKE2 code + room id to the web server in cleartext
- **Where:** `scripts/render-pair-page.mjs:22-27`
- **Issue:** The QR encodes the deep link rewritten from /#p= to /?p= (render-pair-page.mjs:24, mirrored by backend/internal/agent/display.go:31-32 ScanURL). The payload blob is base64url(JSON{r, room, code}) — it contains the SPAKE2 pairing code AND the room id in full (frontend/src/lib/payload.ts:55-59, backend/internal/agent/pair.go:96-103). A query string IS sent to the origin server (unlike a fragment), so for every scanned pairing the web origin, any reverse proxy, CDN, and their access logs receive room id + SPAKE2 code together. base64url is trivially decodable. With both values an attacker who reads those logs (or who is the web-server operator, which in this threat model is NOT one of the trusted two parties) has everything the phone has: they can connect to the relay room and run the SPAKE2 B-side with the exact code, deriving the session key and impersonating the human — there is no second out-of-band secret. ADR 0006 explicitly accepts this trade-off but understates it: the code is not just 'a code an attacker gets one guess at' — it is the complete pairing secret, handed to the server in the clear on every scan.
- **Impact:** Anyone with read access to web-server/proxy logs (or who can observe the ?p= URL in referrer/history/analytics) for a scanned pairing obtains room id + code = the full pairing secret, and can complete SPAKE2 as the phone, derive the session key, and approve/decline as the human. This defeats the 'agent must never be fooled' invariant for the QR path.
- **Fix:** Never place the code in the query. Minimal correct fix: split the QR payload so the query carries only non-secret rendezvous data and the code stays out of any server-observable channel. Concretely: (a) QR/?p= carries only {r, room}; the SPAKE2 code is entered manually on the phone (the manual code is already rendered — render-pair-page.mjs:37, PairingMarkdown display.go:55), so the web origin never sees the code. Update render-pair-page.mjs:24 to build a `?p=` blob WITHOUT `code`, update encodePairPayload/encodePayload to emit a code-less query variant, and have App.tsx require manual code entry when the payload arrives via parseQuery (vs auto-pair only from the #p= fragment). (b) If automatic scan-pairing must keep the code, serve a tiny static shim at the PWA route that, on first byte, copies window.location.search into location.hash and history.replaceState's the query away — and document that the host MUST be configured to not log query strings for that route (defense-in-depth only; the query still hits the wire once). Option (a) is the real fix — it removes the secret from the server-visible path entirely. Also tighten ADR 0006 to state the code is the COMPLETE pairing secret (full SPAKE2 password), not a guess-limited token.
- **Verifier evidence:** Attack path: 1) Agent prints/serves a QR for a scanned pairing; QR encodes `https://web/?p=<base64url(JSON{r,room,code})>` (render-pair-page.mjs:24, display.go:31-33). 2) Phone scans -> GET `https://web/?p=<blob>` hits the static origin / reverse proxy / CDN; the full query string lands in their access logs (standard combined-log format logs the request path+query). 3) Attacker with read access to those logs (or the web-host operator, a non-trusted party per ADR 0004) base64url-decodes the blob: `echo <blob> | base64 -d` -> `{"r":"wss://relay/ws","room":"<16hex>","code":"4F2-9KQ"}`. 4) Attacker dials the relay, joins `room` (rooms-of-two, relay.go:189-205 — they take the B slot before/instead of the phone), and runs SPAKE2 B-side with the exact `code`. Because the code is the full password, key derivation + HMAC key-confirmation both succeed (no second out-of-band secret exists). 5) Attacker is now the "human": secretbox-decrypts requests and sends approve/decline frames the agent will trust. Proof the code is the complete secret, not a guess-limited token: pair.go:39-40 comment "Code is the short pairing code; the SPAKE2 password" and pairAsA uses it directly as the SPAKE2 password (spake2.NewA(code), pair.go:119). Proof the query reaches the server: it's in the HTTP request line of GET `/?p=...`; parseQuery (payload.ts:97-107) is pure client JS that runs post-request. Refutation of ADR mitigation: docs/.../0006:"SPAKE2 limits an attacker who learns the code to a single online guess" — false when the attacker KNOWS the code; the guess-bound is for unknown passwords only.

### [MEDIUM] [security] Relay HTTP server has no ReadHeaderTimeout — slowloris DoS
- **Where:** `backend/internal/relay/relay.go:298-305`
- **Issue:** Serve builds the http.Server with only BaseContext set and an explicit comment disabling read/write timeouts ("No read/write timeouts"). With no ReadHeaderTimeout, an attacker can open many TCP connections to /ws and dribble request-header bytes slowly (or never finish the headers). websocket.Accept (line 108) is only reached after the http stack has read the full request line + headers, so each half-open client ties up a server goroutine/connection indefinitely before any room logic runs.
- **Impact:** Classic slowloris: a handful of slow clients exhaust the server's connection/goroutine budget and starve legitimate pairing — the untrusted network attacker can take the rendezvous offline with trivial resources, no room id needed.

      ReadHeaderTimeout: 10 * time.Second,
      MaxHeaderBytes:    1 << 16, // 64 KiB cap on header size
      BaseContext:       func(_ net.Listener) context.Context { return ctx },
  }

Add "time" to imports. ReadHeaderTimeout bounds only the header phase, so it is compatible with long-lived WebSockets per the existing comment.
- **Verifier evidence:** relay.go:299-305 — srv := &http.Server{Addr, Handler, BaseContext}; comment "No read/write timeouts: WebSocket connections are long-lived." No ReadHeaderTimeout. relay.go:308 — srv.ListenAndServe(). relay.go:108 — websocket.Accept(w, req, acceptOptions) runs only after Handler is invoked (post-header parse). Grep: `rg ReadHeaderTimeout|ReadTimeout|WriteTimeout|IdleTimeout|MaxHeaderBytes` returns no matches in any Go file. Attack path: attacker opens N TCP conns to relay :8080 /ws, sends "GET /ws?room=x HTTP/1.1\r\n" then one header byte every 30s and never sends the terminating CRLF; each conn parks a net/http read goroutine indefinitely with no deadline, exhausting the goroutine/conn budget and starving legitimate pairing. ### [MEDIUM] [security] Pairing code travels in the URL query string (?p=) and is never scrubbed, leaking the SPAKE2 password to server logs / history / Referer
- **Where:** `frontend/src/lib/payload.ts:93-107`
- **Issue:** parseQuery() accepts the pairing payload from location.search (?p=<base64url JSON{r,room,code}>), and App.tsx:92 calls it on load (parseHash(...) ?? parseQuery(window.location.search)). The base64url blob contains the SPAKE2 code, and w = reduce(SHA512(code)) is the ENTIRE offline-guessing defense (crypto.ts:89-91). Unlike the '#p=' fragment (not sent to servers), a query string is: (a) sent in the GET request line to the PWA's web server/CDN -> access logs; (b) exposed as document.location to any third-party script; (c) sent as the Referer header to any cross-origin subresource; (d) written to browser history. Nothing scrubs it: there is no history.replaceState anywhere in src/ (grep confirms zero matches). The docs comment (payload.ts:95) deliberately routes the iOS Camera flow through ?p= because the fragment is dropped. So the documented primary mobile path systematically persists the password where the relay/network/log-readers can recover it, then run SPAKE2 against the room and impersonate the human or the agent -- defeating the 'no relay MITM / offline guessing impossible' invariant.
- **Impact:** Anyone with read access to the PWA server's access logs, a CDN, browser history, or a malicious third-party script on the page recovers the short code, reconstructs w, completes SPAKE2 for that room, and either impersonates the human (forging Decisions) or the agent (phishing the human). The short code is the only secret; once logged it is permanently compromised.
- **Fix:** Scrub the credential from the address bar and history the instant it is parsed, regardless of source, and harden referrer leakage. In App.tsx right after computing initialPayload (around line 92-93), when a payload was found: history.replaceState(null, '', window.location.pathname) — this removes both ?p= and #p= from location and replaces the history entry so it is not retained. Additionally set Referrer-Policy: no-referrer for the PWA (meta tag in src/pages/index.astro or a static _headers/host config) so the code is never sent as Referer to subresources. Note in docs/decisions/architecture/0006 that ?p= still hits the web server access log for the initial GET — that one log line is unavoidable for the iOS-Camera path, so either (a) keep ?p= but ensure the PWA origin does not log query strings / fronts with a no-query-log config, or (b) prefer a server-side 302 that moves p= from query to fragment before the SPA loads. The replaceState fix closes the history/third-party-script/Referer vectors; the access-log vector for the scanned path needs the hosting-side mitigation called out explicitly.
- **Verifier evidence:** Attack path: (1) Human scans agent QR with iOS Camera -> browser opens https://pwa/?p=<base64url{r,room,code}>. (2) That GET is logged by the PWA web server/CDN access log (request line includes ?p=...), emitted as Referer to any cross-origin subresource, and saved to history; any third-party script on the page reads document.location. (3) Log-reader / network observer / malicious script base64url-decodes the blob -> {r, room, code}. (4) Attacker connects to relay r, joins room, and because w=reduce(SHA512(code)) is the only secret (crypto.ts:89, 133, 186-204), runs Handshake.newB(code).start()/finish() to impersonate the human (forge Decisions) or newA to phish — passing HMAC key confirmation (crypto.ts:219-227) since the transcript+w are fully reconstructable. Constraint: relay room is capped at 2 (backend/internal/relay/relay.go logs 'room FULL, rejecting third peer' + conn.Close(StatusRoomFull)), so the attacker must beat the legitimate phone (or act in the pairing window). Grep proof of no mitigation: `rg -n 'replaceState|history\.' frontend/src` -> only App.tsx import lines, zero history scrubbing. parseQuery callers: App.tsx:92 (on-load) and parseScanned (payload.ts:124). buildDeepLink (payload.ts:64) emits '#p=' but render-pair-page.mjs and 0006 route scanned QRs through '?p='.

### [MEDIUM] [security] Deep-link relay URL (payload.r) is used unvalidated to open a WebSocket, enabling UI phishing via a crafted #p= link
- **Where:** `frontend/src/lib/payload.ts:41-53, 68-72`
- **Issue:** validPayload() only checks that r/room/code are non-empty strings; it does not constrain r's scheme or origin. The decoded r flows directly into the transport: session.ts:113 -> RelayClient -> roomURL(relayURL,...) -> new URL(relayURL) (relay.ts:64) -> new WebSocket(url) (relay.ts:60). An attacker who gets a victim to open https://<trusted-pwa>/#p=<attacker-payload> fully controls r, room AND code. The phone auto-starts pairing (App.tsx:88-92), connects to the attacker's relay, and SPAKE2 confirmation SUCCEEDS because the attacker chose the code. The attacker is then the paired 'agent' and can drive the victim's trusted PWA to render arbitrary approval Request cards (title/summary/category) and harvest the human's Decision. E2E confidentiality vs the real relay is not broken, but the trust boundary of 'this is my agent' is: the human approves attacker-supplied prompts inside the legitimate app UI.
- **Impact:** Phishing / social-engineering primitive: a single crafted link makes the genuine PWA show an attacker-authored approval request and exfiltrate the human's yes/no/choice/text answer to attacker infrastructure. Also lets an attacker pin the victim to an attacker-run relay for traffic metadata.
- **Fix:** In validPayload (payload.ts:42-53), constrain the decoded fields before they reach the transport:
- r: parse with new URL(p.r) and require u.protocol === 'wss:' (allow 'ws:' only for localhost) AND, ideally, that u.host matches import.meta.env.PUBLIC_RELAY_URL's host (or an allowlist). Reject otherwise.
- room: require /^[0-9a-f]{16}$/.test(p.room) per the stated contract.
Minimal version:
```
function validPayload(v: unknown): v is PairPayload {
  if (typeof v !== 'object' || v === null) return false;
  const p = v as Record<string, unknown>;
  if (typeof p.r !== 'string' || typeof p.room !== 'string' || typeof p.code !== 'string') return false;
  if (!/^[0-9a-f]{16}$/.test(p.room) || p.code === '') return false;
```

- **Where:** `backend/internal/agent/conn.go (dial), infra/prod/ingress.yaml, infra/base/relay-deployment.yaml:conn.go:52; ingress.yaml:24-30`
- **Issue:** The relay is engineered to keep the room id (the sole long-lived correlator of an agent<->phone relationship) only in RAM, and gates even verbose logging behind AAH_RELAY_VERBOSE (relay.go:28-38). But both peers connect to /ws?room=<id> (conn.go:52 dialWS; relay.ts roomURL), and in prod that request passes through a GKE 'gce' L7 Ingress / Google Cloud Load Balancer (ingress.yaml:14, /ws path 24-30). When GCLB request logging is enabled it records the full request URL including the query string, persisting the room id outside the relay's RAM-only boundary into cloud logs that survive relay restarts. The agent reuses the same room id across reconnects (agent.go:198 reconnect re-dials the same room) and the phone keeps it for the session's life, so a single logged room id correlates the entire relationship's traffic. No annotation disables LB logging for /ws and the relay cannot strip what the LB sees upstream.
- **Impact:** The privacy property 'an attacker who can see the relay learns at most ciphertext + room-id correlation in RAM' is silently widened: the room-id correlator is durably written to cloud LB access logs by the operator's own front door, enabling after-the-fact correlation of which two endpoints (by source IP) talked, when, and how often — exactly the metadata the RAM-only design tried to avoid persisting.
- **Fix:** Prefer (1) as defense-in-depth: carry the room id out of the URL query so it never appears in any HTTP request line/log — send it as a WebSocket subprotocol token (Sec-WebSocket-Protocol header, not in requestUrl) or as the first in-band frame after upgrade; update conn.go:52 dialWS, relay.ts roomURL/handler, and relay.go Handler's query parse. Belt-and-suspenders (2): add `logging: {enable: false}` (or sampleRate 0) to the relay BackendConfig in frontend-config.yaml and document in infra/README that LB request logging for /ws must stay off to honor the RAM-only metadata guarantee.
- **Verifier evidence:** conn.go:52 `u := fmt.Sprintf("%s?room=%s", relayURL, roomID)`; relay.ts:63-65 `roomURL`→`u.searchParams.set('room', roomID)`; relay.go:28-31 logging gated off by default to "preserve the relay's privacy property in prod"; ingress.yaml:14 `gce` class + 24-30 `/ws` → relay through GCLB; frontend-config.yaml:13-28 BackendConfig has no logging stanza; `rg -rn 'logConfig|sampleRate|logging' infra/` → NONE; agent.go:306-308 reconnect re-dials the same `sess.roomID`. Attack path: anyone with Cloud Logging read inspects GCLB access logs; `httpRequest.requestUrl` contains `https://ask-a-human.ai/ws?room=<id>`; correlating identical room ids across log lines reconstructs which two source IPs paired, when, and how often — the exact metadata the RAM-only design avoids persisting.
